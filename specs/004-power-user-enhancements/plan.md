# Implementation Plan: Todo App Power User Enhancements

**Branch**: `004-power-user-enhancements` | **Date**: 2025-12-27 | **Spec**: [spec.md](./spec.md)

## Summary

Implement five power user features for the CLI Todo app: task templates, subtasks with progress tracking, time tracking per task, Pomodoro timer with notifications, and keyboard shortcuts for rapid navigation. All features integrate with the existing Node.js CLI architecture using no new dependencies.

---

## Technical Context

**Language/Version**: Node.js 18+ with ES6 modules
**Primary Dependencies**: inquirer, chalk, luxon, node-notifier, uuid (all existing)
**Storage**: JSON file (`data/tasks.json`) - extended schema
**Testing**: Manual acceptance testing per spec scenarios
**Target Platform**: Cross-platform CLI (Linux, macOS, Windows)
**Project Type**: Single CLI application (phases/phase-1/)
**Performance Goals**: Timer accuracy within 1 second, shortcuts respond within 100ms
**Constraints**: Single active timer at a time, max 20 subtasks per task, TTY required for shortcuts
**Scale/Scope**: Personal productivity tool, single user, local storage

## Implementation Phases

### Phase 1: Task Templates (P1 - Foundation)

**Goal**: Enable saving and loading task templates

**Tasks**:
1. Create `templateService.js` with CRUD operations
2. Extend `tasks.json` schema with `templates` array
3. Add migration in `taskService.js` for new root-level field
4. Add template prompts in `prompts.js`
5. Add menu option 11 in `index.js`
6. Add template display functions in `display.js`

**Dependencies**: None (foundational)

### Phase 2: Subtasks with Nested Checkboxes (P1 - Foundation)

**Goal**: Add subtask support to tasks

**Tasks**:
1. Extend Task schema with `subtasks` array
2. Add subtask methods to `taskService.js`
3. Add subtask prompts in `prompts.js`
4. Add subtask display (indented, progress) in `display.js`
5. Extend task update flow in `index.js`
6. Handle parent completion cascade

**Dependencies**: None (foundational)

### Phase 3: Time Tracking per Task (P2)

**Goal**: Track time spent on tasks

**Tasks**:
1. Create `timerService.js` with start/stop/recover
2. Extend Task schema with `timeEntries`, `totalTimeMs`
3. Add `activeTimerState` to root schema
4. Add timer prompts in `prompts.js`
5. Add menu option 12 in `index.js`
6. Add timer display (active, total) in `display.js`
7. Handle orphaned timer on startup
8. Handle SIGINT/SIGTERM for timer save

**Dependencies**: Phase 1, 2 complete (shared schema migration)

### Phase 4: Pomodoro Timer (P2)

**Goal**: Implement Pomodoro technique timer

**Tasks**:
1. Extend `timerService.js` with Pomodoro functions
2. Add `pomodoroConfig` to root schema
3. Add Pomodoro prompts in `prompts.js`
4. Add menu option 13 in `index.js`
5. Add Pomodoro display (countdown, count) in `display.js`
6. Integrate with `notificationService.js` for alerts
7. Handle daily count reset

**Dependencies**: Phase 3 complete (timerService base)

### Phase 5: Keyboard Shortcuts (P3)

**Goal**: Enable keyboard-driven navigation

**Tasks**:
1. Create `keyboardService.js` with readline integration
2. Add TTY detection and graceful fallback
3. Register shortcuts for common actions
4. Add help overlay display in `display.js`
5. Integrate with main menu loop in `index.js`
6. Disable shortcuts during prompt input

**Dependencies**: Phases 1-4 complete (all menu options exist)

### Phase 6: Polish & Integration

**Goal**: Final integration and documentation

**Tasks**:
1. Update README.md with Phase 4 features
2. Test all features end-to-end
3. Validate quickstart.md scenarios
4. Handle edge cases per spec
5. Performance validation (timer accuracy, shortcut response)

**Dependencies**: Phases 1-5 complete

## Risk Mitigations

### Risk 1: Timer Persistence

**Risk**: User loses tracked time on crash
**Mitigation**: Save timer start time immediately; recover on startup

### Risk 2: Keyboard Shortcuts Compatibility

**Risk**: Shortcuts don't work in non-TTY environments
**Mitigation**: Detect `process.stdin.isTTY`; graceful fallback to menu

### Risk 3: Schema Migration

**Risk**: Existing data incompatible with new schema
**Mitigation**: All new fields have defaults; migration runs on load
